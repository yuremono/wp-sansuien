# Anonymize

旧案件を特定できる固有名詞と識別情報を洗い出し、表示文脈を保ちながら案件名へ統一する。画像内の名称はテーマ名と分離し、ユーザー指定または同一文字数の架空名へ統一する。

## 最優先ルール

- 作業先の `AGENTS.md` と既存命名規則を優先する。
- `.bak` は読込・編集・削除しない。
- ユーザーの未コミット変更を保持する。
- 置換対象、テーマ名、画像用架空名が判断できない場合だけ確認する。
- スキル作成や手順書更新を同時に依頼された場合は、実作業と検証を完了してから更新する。
- 画像は変更が必要な箇所をリストアップしスキル完了後にユーザーの指示を仰ぐ。勝手に `imagegen` スキルを使わない。

## 1. 読み取り専用調査

1. `git status --short` で既存変更を確認する。
2. PHP、HTML、JavaScript、CSS、JSON、XML、Markdown、シェル、画像実体を列挙する。
3. 店名の表記揺れ、読み仮名、ローマ字、text domain、PHP package、CSS class、定数、メニュー名、URLを大文字小文字を変えて検索する。
4. 人名、地名、住所、電話番号、実在企業、酒造、商品・銘柄、SNS投稿URL、地図埋め込みを検索する。
5. ファイル名とディレクトリ名にも旧案件名がないか確認する。
6. `file` で画像実体を確認する。拡張子が画像でも404 HTMLが保存されている場合がある。
7. テンプレートとCSSから実際に参照される画像を特定する。

検索語は既知の名称だけに限定しない。DOMの見出し、商品一覧、alt、title、meta、構造化データ、コメント、管理画面初期値も確認する。

## 2. 置換方針

- DOMに出る旧店名・ブランド名は、ユーザー指定のテーマ表示名へ置換する。
- text domain、slug、package、独自DOM classなどコード用識別子は、案件の英字名と既存の大小文字規則へ合わせる。
- 住所と電話番号はテーマ名を無理に入れず、表示用途に合う明確なダミー値へ置換する。
- 住所に関しては、実在しない架空の住所に置き換え、iframeは東京都全域など適度に抽象化する
- 実在する人名、企業、製造元、商品・銘柄、SNS投稿URL、地図座標は旧案件を特定できない内容へ変更する。
- 一般名詞、料理名、酒類、技術ライブラリ、ライセンスURLは固有名詞と誤認して置換しない。
- テンプレート基盤の `example` は旧案件名ではない限り維持する。
- 置換前後の対応表を頭の中または作業メモで一貫させ、同じ対象に複数の置換語を使わない。

## 3. テキストとファイル名

1. 表記揺れを長い文字列から短い文字列の順で置換する。
2. ファイル名を変更した場合はPHP、CSS `url()`、JavaScript、ビルド・デプロイ設定の参照も同時に変更する。
3. 機械置換後に周辺文を必ず読む。
4. 「居酒屋にある居酒屋」「居酒屋から一文字取った」のような不自然な文章は、テーマの説明として自然な文へ書き直す。
5. 商品一覧では商品名だけを置換し、価格、度数、カテゴリなど一般情報を壊さない。
6. 外部SNS投稿は必要なら `#` など無害なリンクへ変更し、地図iframeは旧所在地を表示しない。

## 4. 画像

変更が必要な箇所をリストアップしスキル完了後にユーザーの指示を仰ぐ。

1. ロゴ、看板、ボトルラベル、SNS画像など、固有名詞が視認できる画像を確認する。

以下は生成指示があった場合のみ読む

- 画像生成する場合は画像内だけで使う架空名を一つ決め、全生成画像で統一する。
- 元名称と同じ文字数を求められた場合は、漢字・ひらがななど表記単位ごとに文字数を合わせる。
- 生成結果の文字が正確か目視する。誤字、別名、欠けがあれば再生成する。
- 元画像ごとの縦横比とピクセル寸法を取得し、`resize` と中央cropで維持する。
- 参照中の各ファイルへ差し替え、プロジェクト外の生成先だけに成果物を残さない。
- 料理・内装など固有名詞がない写真は変更しない。

## 5. 検証

1. 旧店名、読み、ローマ字、住所、電話、企業、人名、地名、商品名、旧URLを再検索し、残存ゼロを確認する。
2. PHP lint、PHPCS、`git diff --check`、既存テストを実行する。
3. ローカルWordPressをブラウザでデスクトップ・モバイル表示する。
4. トップ、商品一覧、店舗情報、SNS表示を確認する。
5. ロゴ文字、商品ラベル、画像crop、alt、見出し、電話リンク、外部リンクを確認する。
6. コンソールエラー、404、レイアウト崩れを確認する。

## 実作業で判明した注意点

- 画像拡張子のファイルがHTMLの場合、画像生成で上書きする前に参照と用途を確認する。
- 単純な全文置換は固有名詞を消せても文章を壊すため、表示文の校正を独立工程にする。
- 実在銘柄は店名検索だけでは残る。商品名欄とSNS本文を別に抽出する。
- 同じ生成画像を複数寸法へ展開するときも、各元ファイルの寸法を個別に維持する。
- ロゴ用の横長画像を小型ヘッダーへ縮小すると文字が潰れやすい。実表示サイズで確認する。
- SNS画像と投稿リンクの両方を処理しないと、旧案件への導線が残る。
- 住所文字列を消してもGoogle Mapsの埋め込みURLに座標と住所が残る。

## 完了条件

- 旧案件を特定できる固有名詞と識別情報が、ファイル名、ソース、DOM、メタデータ、URLから除去されている。
- テーマ表示名とコード識別子が案件規則へ統一されている。
- 置換後の文章が自然で、表示とリンクが壊れていない。
- コード検証とブラウザ確認が完了している。
